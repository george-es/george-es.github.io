专题素材管理项目,我学会了什么?

发表于 2021-09-22 分类于 项目经验 阅读次数： 本文字数： 14k 阅读时长 ≈ 24 分钟

专题素材管理项目第一阶段快结束了，已经到了测试环节，回顾下这个项目，用了那些知识

背景

随着公司业务发展，多品牌页面、sheinX、大小促销活动均有专题页面承载，专题的配置工作量占比逐渐升高。

搭建页面框架 → 素材准备（图片&多语言）→ 父专题配置 → 子专题配置

痛点

问题：设计改图工作量大，每个语种的专题需要进行对应的语种图片修改，耗时且易出错。

预期：设计可以脱离图片小语种替换工作

问题：专题配置工作量大，全市场专题小语种数量较多，共有16个小语种，需要对每个语种的专题进行各组件图片替换及检查，耗时且易出错

预期：提高专题配置效率，避免重复性工作

技术思考

功能上

这里贴个测试用例

设计上

从功能上，我们可以看出，项目的入口在素材管理，两个核心页，图片素材管理页和 PSD 编辑页，文案管理页，是通过 iframe 嵌入第三方的页面，有两个搜索框，分别支撑各自的列表页，也就是素材列表和图片组列表，PSD 编辑部分，为了流畅度和性能，生成图片组之前，全都操作状态都在前端维护。

一句话描述产品

专题素材管理系统，是为了提升运营配置专题效率而诞生的，对于同样的图片，要根据国家去批量替换语言，譬如说，图片中有行文字，叫中国，到了美国站，则显示 China

技术分析

图层

它的复杂点在于 PSD 文件解析，在这里，我们用的是 psd.js 库来处理，它能将 psd 文件的每个图层信息解析出来，在渲染方面，用到的是 konva 库，我们要做的是将 PSD 解析出来的图层信息，渲染到 konva 上。

路由

该项目分3个页面，分别是素材配置页，图片组配置页，PSD 操作页，需要对路由进行操作的组件，通过 react-router-dom 库的 withRouter 设置

其中 match 用于动态路由，里面存放着路由属性的参数字段

history 对象有个各种路由操作的方法

  • push(url, params) 跳转到 url 页面，其中 params 是传递跳转路由的参数，它被存放在 location 的 state 里面
  • goBack()：回到上一个页面

状态管理

状态管理用的是 redux，异步用的是 redux-saga，封装上，为了让 redux 的书写变得更容易管理，和更加优雅，我们引用了 @reduxjs/toolkit 库，这个库详细请看 Post not found: Redux ToolKit 改造项目，对搜索的列表结果内容，都存储在 redux 中，对初始化数据进行分类，划分素材管理，图片组，psd编辑内容，以及公共数据

由于 saga 特性，用的是 Generator 生成器，它无法将接口的返回值直接抛出，在一些场景下，譬如说下一个接口要依赖上一个接口的值，一般这种情况有两个解决方案

  • 将数据存到 redux 中通过 useEffect 监听它的变化

  • 使用回调函数

    逻辑就是，将回调函数作为参数，传入。

组件架构设计

基于对项目的分析，我把它分成两大类业务组件和公共组件，其中公共组件包括，上传，动态创建输入框，表单渲染，以及一些自定义 Hooks 等，业务组件也就是根据业务需要设计的，它依托公共组件。

其中 lists 是输入框内容列表，在这里指，素材包名称，素材包ID，投放站点，创建人等

我是这样设计搜索框组件 lists 的数据结构

这只是公共组件，根据公共组件还要设计相应的业务组件

根据这两个搜索框的业务性质，我们可以设计两个业务组件，像投放站点，语言，创建人这些下拉选项框的内容，都是通过 API 接口获取的。

因此这两个业务组件，它们只处理数据，不负责 UI 渲染

PSD 处理

以往的图片处理，都是通过接口上传到服务器上，服务器返回一个图片地址，前端去读取，由于 PSD 文件小到几十兆，上到几百兆，如果也按照这样的设计思想，用户等待的时间是非常长的，也不是很合理，因此，PSD 的操作就不走后端了，纯前端。

状态管理

PSD 功能模块也不是放在一个组件中就可以完成了，拆分了多个模块，模块与模块之间的交互就会涉及到状态管理，前端的状态管理有 3 种方式，sessionStorage、redux、window，由于前两个对存储大小都有限制，因此我选择存储在 window 上，需要注意的是，操作结束了，记得释放就好了。

PSD 解析

通过 URL.createObjectURL 处理的文件，返回 Blob 类型，若想使用文件，需要进行转换，这就是所谓的间接存储
